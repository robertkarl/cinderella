# Token Introspection for MCP Servers

How an MCP server running inside Claude Code can know the conversation's token count — without the model self-reporting it.

## The Problem

Glass Slipper's companion window shows a savings dashboard: "you saved $X by running this locally instead of on Opus." To calculate savings, we need to know the conversation's context size (token count) at the time of each MCP tool call. Bigger context = more expensive cloud call = more savings.

The MCP protocol has no built-in mechanism to pass conversation metadata (like token counts) alongside `tools/call` requests. The model can't reliably self-report its own token count. The `/usage` and `/context` commands show data to the user but don't pipe it back into the model's context.

## The Solution: Read Claude Code's Session JSONL

Claude Code writes every assistant response — with full usage data — to session JSONL files at:

```
~/.claude/projects/<project-path-hash>/*.jsonl
```

Each assistant message includes a `usage` object:

```json
{
  "type": "assistant",
  "message": {
    "usage": {
      "input_tokens": 45231,
      "output_tokens": 1892,
      "cache_read_input_tokens": 38000,
      "cache_creation_input_tokens": 7231
    }
  }
}
```

### What each field means

| Field | Meaning |
|---|---|
| `input_tokens` | Total input tokens for that turn (includes cache reads + cache creation) |
| `output_tokens` | Tokens generated by the model |
| `cache_read_input_tokens` | Input tokens served from Anthropic's prompt cache |
| `cache_creation_input_tokens` | Input tokens written into the cache on this turn |

The `input_tokens` value on the most recent assistant message is effectively the conversation's current context size — exactly what we need for savings calculations.

### How to read it

1. Find the most recently modified `.jsonl` in `~/.claude/projects/`
2. Read the last line where `type == "assistant"` and `message.usage` exists
3. Extract `input_tokens` as the context size

The gauntlette project's `estimate-tokens.sh` (at `~/Code/gauntlette/skills/bin/estimate-tokens.sh`) is a reference implementation that does exactly this with jq.

### Rough jq one-liner

```bash
jq -s '[.[] | select(.type == "assistant" and .message.usage != null)] | last | .message.usage.input_tokens' \
  "$(find ~/.claude/projects -name '*.jsonl' -type f -exec stat -f '%m %N' {} + | sort -rn | head -1 | sed 's/^[0-9]* //')"
```

## Implications for Glass Slipper

The `context_tokens` parameter on MCP tool schemas can be made **optional** (or removed entirely). Instead, the Rust MCP server can:

1. On each tool call, find and read the active Claude Code session JSONL
2. Grab `input_tokens` from the last assistant message
3. Use that for the savings calculation in `ActivityLogger`

This is more accurate than asking the model to estimate, and removes friction from the tool call interface.

## Caveats

- Session JSONL path uses a hash of the project directory — need to resolve the correct file
- The file is append-only and can grow large; read from the end, not the beginning
- There may be a brief race between Claude writing the usage data and the MCP server reading it (the tool call happens mid-turn)
- This approach is specific to Claude Code; other MCP hosts won't have this file
