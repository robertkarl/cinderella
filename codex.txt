• I searched current MCP security discussions across official docs, GitHub advisories, HN, and security blogs. Realistically: people do not “solve” MCP security. They reduce blast radius and treat MCP like a plugin/RCE surface.

  The Practical Consensus

  1. Treat MCP servers as installed software, not harmless API metadata.
     The official MCP docs explicitly warn that local MCP servers can have direct access to the user’s machine and should be sandboxed, consented, and restricted. GitHub’s modelcontextprotocol/servers security page also says the reference servers are educational examples, not production-ready
     implementations.
  2. Do not expose generic shell tools unless you are comfortable with arbitrary code execution.
     OX Security’s 2026 disclosure argues the STDIO/config execution model has caused real RCE issues across MCP-enabled products. Whether you call it “by design” or a vulnerability, the practical control is the same: never let untrusted input influence a command, and do not run broad shell tools
     outside a sandbox.
  3. Least privilege is the main defense.
     HN discussions around the GitHub MCP exploit repeatedly converge on this: if the agent can read private repos and write to public repos in one session, prompt injection can bridge those worlds. Invariant’s GitHub MCP writeup recommends policies like “one repo per session” and runtime dataflow
     controls.
  4. Human approval helps, but “Always Allow” defeats most of it.
     Simon Willison points out that MCP clients should show exposed tools, show invocations clearly, and prompt before actions. His practical position is to treat those “SHOULD” recommendations as “MUST.” But approval dialogs only work if the UI shows the real risk and users do not blanket-
     approve.
  5. Tool descriptions are part of the attack surface.
     Invariant’s tool-poisoning research shows malicious or changed tool descriptions can influence calls to other trusted tools. The mitigations people use are tool/version pinning, displaying AI-visible instructions, hashing tool descriptions, and warning on changes.
  6. Use proxies/scanners/guardrails, but do not rely on them alone.
     Tools like Invariant MCP-Scan and Trail of Bits mcp-context-protector add scanning, pinning, quarantines, ANSI sanitization, and runtime monitoring. Useful, but they are compensating controls, not a substitute for sandboxing and least privilege.
  7. For enterprises, MCP becomes an inventory and governance problem.
     The realistic enterprise posture is: approved server registry, signed/pinned versions, scoped OAuth tokens, no long-lived secrets, audit logs, egress controls, sandboxed execution, and security review before adding new MCP servers.

  Applied To Glass Slipper
  For this repo, I would not ship the current MCP shell-command tools as-is. The realistic secure path is:

  - remove generic shell execution, or route it through the existing classifier/sandbox/timeout/output caps;
  - make tools purpose-specific, like summarize_build_output, not run arbitrary command;
  - run the MCP server with minimal filesystem and network access;
  - pin bundled binaries and validate installed MCP config paths;
  - block SSRF/private-network fetches by default;
  - log every tool call with arguments and result size;
  - never rely on model alignment to enforce security boundaries.

  Useful sources: official MCP security docs, especially local server compromise and SSRF guidance; GitHub MCP server advisories; Invariant’s GitHub MCP and tool-poisoning research; Simon Willison’s MCP prompt-injection post; Trail of Bits’ MCP security wrapper; and the HN threads on GitHub MCP
  exploitation and “The S in MCP Stands for Security.”
